# GitHub Sentinel

[English](README-EN.md) | 简体中文

GitHub Sentinel 是一款开源工具类 AI Agent，专为开发者和项目管理人员设计。它能够定期获取并汇总订阅的 GitHub 仓库最新动态，围绕订阅管理、更新检索、通知系统、报告生成和多模型摘要形成完整的项目跟踪工作流。

通过及时获取和推送仓库更新，GitHub Sentinel 帮助团队提升协作效率和项目管理便捷性，使用户能够更高效地跟踪项目进展，快速响应和处理变更，确保项目始终处于可观察、可追踪、可同步的状态。

## 1. 项目定位

GitHub Sentinel 聚焦“仓库订阅、更新检索、通知推送、报告生成”这一类开发协作场景。

它不是单纯的仓库浏览工具，而是一个面向团队协作的仓库动态哨兵：用户订阅关心的仓库后，系统负责检索更新、整理信息、生成报告，并为通知分发提供统一入口。

### 1.1 面向用户

- 开发者：集中跟踪依赖项目、开源项目或团队仓库的更新。
- 项目管理人员：了解项目进展、关键提交和近期变化。
- 技术负责人：掌握多个仓库的变更节奏和协作状态。
- 团队成员：通过报告快速了解项目动态，减少人工同步成本。

### 1.2 解决的问题

- 仓库动态分散，人工查看成本高。
- 多仓库项目进展难以及时汇总。
- 提交记录、Issue、Pull Request、Release 等信息缺少统一摘要。
- 团队对项目变化的响应依赖人工提醒。
- 日报、周报、项目进展报告整理成本高。

## 2. 系统能力

### 2.1 订阅管理

订阅管理用于集中维护用户关注的仓库，帮助用户把分散的项目统一纳入跟踪范围。

系统能力包括：

- 通过仓库地址添加关注仓库。
- 集中查看已订阅仓库。
- 支持公开仓库和私有仓库两类订阅场景。
- 支持为私有仓库配置访问凭证。
- 支持按仓库维度管理订阅状态。
- 支持移除不再关注的仓库。
- 订阅信息会隐藏敏感凭证，只展示是否已配置。

### 2.2 更新检索

更新检索用于持续获取订阅仓库的最新动态，让团队及时掌握项目变化。

系统能力包括：

- 获取订阅仓库的最新动态。
- 汇总提交记录、Issue、Pull Request 等项目更新。
- 识别一次代码推送中的多条提交。
- 提取每条提交的说明信息。
- 减少重复动态对报告内容的干扰。
- 支持公开仓库和私有仓库的更新检索。
- 为报告生成和通知推送提供统一的数据来源。

### 2.3 通知系统

通知系统用于将项目最新进展及时推送给订阅者。

系统能力包括：

- 为订阅配置通知目标。
- 在生成报告后向订阅者同步项目进展。
- 通知内容围绕仓库更新、提交记录和项目报告组织。
- 支持邮件、Webhook、企业协作工具等通知形态。
- 让团队成员无需频繁主动查看仓库，也能获得关键更新。

### 2.4 报告生成

报告生成用于把仓库更新整理成适合团队阅读和传播的项目进展材料。

系统能力包括：

- 根据仓库动态生成项目进展报告。
- 在报告中呈现关键提交说明。
- 支持查看历史报告。
- 支持按时间范围查看报告，例如最近 1 小时、24 小时、7 天或 30 天。
- 支持日报、周报、版本摘要、项目进展摘要等报告场景。
- 支持 Markdown、HTML、邮件正文等报告输出形式。

### 2.5 多模型支持

多模型支持用于提升报告的可读性和信息整理质量，让项目报告更接近自然语言总结。

系统能力包括：

- 支持内置模型能力，用于生成自然语言项目报告。
- 支持用户自行配置模型接入方式。
- 支持接入当前主流模型服务。
- 支持配置模型服务地址、模型名称、访问密钥和请求参数。
- 支持按不同报告场景选择不同模型。
- 支持将提交记录、Issue、Pull Request 等内容整理成自然语言摘要。

### 2.6 定时任务

定时任务用于让仓库更新按用户配置的频率自动执行，减少人工触发成本。

系统能力包括：

- 支持秒级任务频率配置。
- 支持每日、每周等常见周期。
- 支持长期运行的任务模式。
- 支持不同仓库使用不同更新频率。
- 支持自动获取更新并生成后续报告。
- 支持记录任务执行状态，便于排查异常。

### 2.7 图形化界面

图形化界面用于降低工具使用门槛，使非后端开发人员也能完成订阅、检索和报告操作。

系统能力包括：

- 提供可视化操作入口。
- 支持通过表单完成订阅管理。
- 支持触发仓库更新检索。
- 支持查看报告结果。
- 降低命令行和接口使用门槛。
- 适合快速完成轻量化项目管理操作。

### 2.8 容器化部署

容器化用于降低部署成本，提升跨环境运行一致性。

系统能力包括：

- 支持通过容器方式部署服务。
- 支持在不同服务器环境中快速启动。
- 支持通过环境变量管理运行配置。
- 支持把应用、配置和运行环境标准化。
- 降低迁移、部署和交付成本。

### 2.9 持续集成

持续集成用于保障项目稳定性和交付质量。

系统能力包括：

- 使用自动化测试覆盖核心流程。
- 使用代码检查保障基础质量。
- 支持在发布前执行质量校验。
- 支持围绕订阅、检索、报告和通知建立回归验证。
- 为稳定交付和团队协作提供质量基础。

### 2.10 安全与数据保护

安全与数据保护用于降低敏感信息泄露风险，让仓库凭证和用户配置更安全。

系统能力包括：

- 访问凭证可按需填写。
- 公开仓库无需配置访问凭证。
- 私有仓库凭证会加密保存。
- 页面和接口不会展示凭证明文。
- 错误提示避免暴露内部敏感细节。
- 敏感配置通过运行环境注入。

## 3. 核心工作流

```text
创建订阅
  -> 配置仓库地址与访问令牌
  -> 定时或手动检索仓库最新动态
  -> 整理仓库更新内容
  -> 过滤重复动态
  -> 生成项目进展报告
  -> 通知订阅者或进入报告查看流程
```

这个流程让仓库变化从“分散事件”变成“可阅读报告”，再进一步支撑团队同步和项目管理。

## 4. 使用场景

### 4.1 开源项目跟踪

开发者可以订阅关注的 GitHub 开源仓库，集中查看近期提交和项目动态。

### 4.2 团队项目管理

项目负责人可以通过报告了解团队仓库的近期更新，减少逐个仓库查看的时间成本。

### 4.3 私有仓库监控

企业团队可以为私有仓库配置访问令牌，生成内部项目进展摘要。

### 4.4 日报与周报整理

团队可以按时间范围查看报告，用于整理日报、周报或项目状态同步材料。

## 6. 快速开始

### 6.1 安装依赖

本项目使用 `uv` 管理 Python 版本、虚拟环境和依赖。

```powershell
uv sync --group dev
```

### 6.2 创建本地数据库文件

首次启动前，需要在项目根目录手动创建 SQLite 数据库文件：

```powershell
New-Item -ItemType File -Path .\github_sentinel.db -Force
```

应用启动后会自动创建缺失的数据表。

### 6.3 启动服务

```powershell
uv run python -m uvicorn app.main:create_app --factory --reload
```

启动后可访问：

| 页面/接口  | 地址                                 |
| ------ | ---------------------------------- |
| 服务首页   | `http://127.0.0.1:8000/`           |
| API 文档 | `http://127.0.0.1:8000/docs`       |
| 健康检查   | `http://127.0.0.1:8000/api/health` |

如果在 Windows 下执行 `uv run uvicorn app.main:create_app --factory --reload` 出现 `Failed to canonicalize script path`，请使用上面的 `uv run python -m uvicorn ...` 启动方式。

### 6.4 配置说明

项目默认读取 `config/settings.toml` 中的非敏感配置。

敏感信息应通过环境变量注入，不建议写入配置文件或提交到仓库。

| 配置项                      | 作用        |
| ------------------------ | --------- |
| `DATABASE_URL`           | 数据库连接地址   |
| `TOKEN_ENCRYPTION_KEY`   | 访问令牌加密密钥  |
| `GITHUB_SENTINEL_CONFIG` | 自定义配置文件路径 |

生产环境建议显式配置 `TOKEN_ENCRYPTION_KEY`。可使用以下命令生成 Fernet key：

```powershell
uv run python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```
